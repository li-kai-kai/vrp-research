# 汶川案例模型输入数据来源与表格提取

> 本文档整理自论文 PDF `s10479-018-3037-2.pdf` 中的汶川地震案例。重点是“模型输入数据来自哪里”和“案例中直接使用了哪些表格数据”。

## 数据来源汇总

论文中的汶川案例不是凭空构造的随机网络，而是把真实灾区节点、道路、需求、车辆和受损信息整理成模型输入。各类数据来源如下：

| 数据类别 | 具体内容 | 原始数据来源或估计方法 |
|---|---|---|
| 需求节点需求量 | 各乡镇/节点的救援物资需求量 `D`；结合车辆容量单位，合理解释为吨 | 根据《四川统计年鉴 2008》中的人口数据，以及房屋倒塌比例估计 |
| 房屋倒塌比例 | 用于估计各需求节点受灾程度和需求量 | 可由卫星影像、红外技术、航拍技术预测 |
| 道路通行时间 | 节点之间道路/链路的行驶时间 | Google Earth |
| 道路容量 | 路段基础通行能力 `ca`，用于 BPR 通行时间函数 | 论文汶川案例未逐条给出；本文按路段通行时间、供给节点连接关系和山区道路场景补充估算 |
| 受损道路长度 | 用于估计受损节点修复时间 | 红外技术或航拍技术识别受损道路/链路长度 |
| 修复时间 `sd` | 每个受损节点所需修复时间 | 受损道路长度除以维修队能力；论文假设维修队能力为 `0.5 km/h` |
| 车辆数量 | 各类应急救援车辆数量 | 四川省交通运输厅公路局 |
| 车辆占用 OD | 不同车辆类型对 OD 流量的占用 | 四川省交通运输厅公路局 |
| 维修工作站/供给节点 | 都江堰、彭州、什邡；供给量 `S` 结合车辆容量单位，合理解释为吨 | 案例设定，作为维修队出发点和物资供给点；供给量原始来源未说明 |
| 维修队信息 | 维修队所属工作站、初始出发位置、维修队数量 | 固有资源属性，应来自应急抢修队伍台账或案例设定；模型决定的是派工顺序，不决定队伍本身是否存在 |
| 维修能力 | 每支维修队的道路修复能力，案例中为 `0.5 km/h` | 固有资源属性或情景参数；论文假设各维修队能力相同 |
| 规划周期 | 灾后黄金 72 小时，案例中 `η = 8h` | 论文模型设定 |

## 表 2：受损道路/节点与修复时间数据

该表是上层道路修复排班模型的关键输入。论文将受损道路/链路抽象为受损节点，`sd` 表示修复时间，单位为分钟。

| 8 | 47 | 18(120), 25(60) | 270 |
| 9 | 57 | 22(12), 24(16) | 810 |
| 10 | 66 | 24(28), 30(36) | 270 |
| 11 | 71 | 26(16), 28(16) | 720 |
| 12 | 72 | 26(30), 34(16) | 1050 |
| 13 | 76 | 28(16), 29(12) | 360 |
| 14 | 87 | 33(30), 34(30) | 180 |
| 15 | 92 | 34(40), 35(60) | 450 |
| 16 | 98 | 36(32), 37(24) | 330 |

说明：

- **受损节点编号**：论文把一段受损道路抽象成一个“受损节点”，方便在模型里安排维修队访问和修复。
- **对应道路/链路编号**：该受损点所在的道路编号。道路编号由表 1 的道路连接关系确定，例如 link ID 1 是节点 1 到节点 4，其他编号依此类推。
- **道路两端节点及通行时间**：括号外是道路两端的节点编号，括号内是该端到受损点的通行时间。例如 `1(20), 17(8)` 表示该受损点位于节点 1 与节点 17 之间，从节点 1 到受损点约 20 分钟，从节点 17 到受损点约 8 分钟。
- **修复时间 `sd`**：维修队完成该受损道路修复所需时间，单位为分钟。
- 修复时间由受损道路长度除以维修队能力估算；受损道路长度可由红外技术或航拍技术识别。
- 论文假设每支维修队能力相同，均为 `0.5 km/h`。

通行时间口径说明：本表中的通行时间是“道路端点到受损位置”的接近时间，主要用于计算维修队到达受损点的路由成本。它不是表 1 中两个地点节点之间的完整路段基础通行时间。

## 数据完整性检查

对照论文模型和汶川案例，当前文档已经提取或整理了以下输入：

| 数据项 | 当前状态 | 备注 |
|---|---|---|
| 供给节点、需求节点、节点连接关系 | 已提取 | 来自节点与道路连接表 |
| 供给量 `S`、需求量 `D` | 已提取 | 结合车辆容量，按吨解释 |
| 基础通行时间 `t0` | 已提取 | 表 1 中节点到相邻节点的分钟数 |
| 道路容量 `ca` | 已补充估算 | 论文案例未逐条给出，当前为场景假设 |
| 受损道路/受损节点 | 已提取 | 来自受损节点表 |
| 受损点接近时间 | 已提取 | 表 2 中端点到受损位置的分钟数 |
| 修复时间 `sd` | 已提取 | 单位为分钟 |
| 维修队工作站和初始节点 | 已整理 | 由案例设定和表 13 路径整理 |
| 维修能力 | 已整理 | `0.5 km/h` |
| 车辆类型、容量、数量、占用 OD | 已提取 | 来自车辆信息表 |
| 规划期和更新周期 | 已整理 | 72 小时、`η = 8h`、9 个周期 |

仍然没有从论文中获得逐项原始数据的部分：

| 缺口数据 | 是否能从论文直接提取 | 处理建议 |
|---|---|---|
| 每条道路的真实容量 `ca` | 不能 | 当前用容量估算表补齐；若有道路等级、车道数或交通部门容量数据，应替换估算值 |
| 普通交通 OD 需求 `Qrs` | 不能 | 当前版本可暂不纳入；若后续需要完整复现论文中“普通交通 + 救援交通”共同作用的可达性机制，再补充交通调查数据或构造情景 OD |
| 每条路段的普通交通流 `fa` | 不能 | 当前版本可暂不单独建模；若后续恢复普通交通层，则可由 `Qrs` 和路径分配计算得到 |
| 每个需求点房屋倒塌比例 | 不能 | 论文只说明可由卫星、红外、航拍预测，没有列出逐节点比例 |
| 每段受损道路长度 | 不能 | 论文只列出换算后的修复时间 `sd`，没有列出原始受损长度 |
| 供给量 `S = 3000` 的原始仓储来源 | 不能 | 论文给出案例值，但未说明来自仓库台账还是案例设定 |
| 算法参数 | 可从正文整理 | 如种群规模、迭代次数、交叉/变异概率等；它们不是汶川案例地理输入，但复现实验需要 |

结论：如果目标是复现论文中的汶川案例核心输入并先跑通“道路修复 + 救援物流”主线，当前文档已经覆盖了论文附录直接给出的核心输入表，并补充了维修队表和道路容量估算表。当前阶段可以暂不纳入普通交通 OD `Qrs` 及其对应路段交通流 `fa`；若后续目标升级为完整复现论文中的综合可达性机制，再补充普通交通层相关数据。
